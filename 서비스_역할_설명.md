# GitHub, Vercel, Supabase 역할과 목적

## 📚 전체 구조 개요

```
[개발자] 
    ↓ 코드 작성
[GitHub] ← 코드 저장소 (버전 관리)
    ↓ 코드 푸시
[Vercel] ← 웹사이트 배포 (호스팅)
    ↓ API 요청
[Supabase] ← 데이터베이스 & 백엔드 서비스
```

---

## 1. GitHub (코드 저장소)

### 🎯 역할
**코드 버전 관리 및 협업 플랫폼**

### 📋 주요 기능

#### ✅ 코드 저장
- 프로젝트의 모든 소스 코드를 저장
- 파일 변경 이력 관리
- 여러 버전의 코드를 추적

#### ✅ 버전 관리
- 코드 변경 사항 추적
- 이전 버전으로 되돌리기 가능
- 변경 이력 확인

#### ✅ 협업
- 여러 개발자가 동시에 작업
- 코드 리뷰 및 피드백
- 이슈 관리 및 프로젝트 관리

#### ✅ 백업
- 클라우드에 코드 저장
- 컴퓨터가 고장나도 코드 보존

### 💡 실제 사용 예시
```
1. 로컬에서 코드 작성
2. git add . (변경사항 추가)
3. git commit -m "메시지" (저장)
4. git push (GitHub에 업로드)
```

### 🔗 우리 프로젝트에서의 역할
- `kwangdong-site` 프로젝트 코드 저장
- `vercel.json`, `package.json` 등 설정 파일 관리
- 코드 변경 이력 추적

---

## 2. Vercel (웹사이트 호스팅)

### 🎯 역할
**웹사이트를 인터넷에 배포하는 플랫폼**

### 📋 주요 기능

#### ✅ 웹사이트 호스팅
- React, Vue, Next.js 등 프론트엔드 프레임워크 배포
- 정적 웹사이트를 인터넷에 공개
- 전 세계 사용자가 접속 가능

#### ✅ 자동 배포
- GitHub에 코드를 푸시하면 자동으로 배포
- 수동 배포 불필요
- 빠른 업데이트 가능

#### ✅ HTTPS 자동 적용
- 보안 연결 자동 설정
- SSL 인증서 자동 관리

#### ✅ CDN (Content Delivery Network)
- 전 세계 서버에 콘텐츠 분산
- 빠른 로딩 속도
- 사용자 위치에 가까운 서버에서 제공

#### ✅ 환경 변수 관리
- API 키 등 민감한 정보 안전하게 관리
- 프로덕션과 개발 환경 분리

### 💡 실제 사용 예시
```
1. GitHub에서 코드 가져오기
2. npm run build (빌드)
3. dist 폴더를 인터넷에 배포
4. https://kwangdong-site.vercel.app URL 생성
```

### 🔗 우리 프로젝트에서의 역할
- 광동제약 방문예약 웹사이트를 인터넷에 배포
- 사용자가 브라우저로 접속할 수 있게 함
- Supabase와 통신하는 프론트엔드 제공

---

## 3. Supabase (백엔드 서비스)

### 🎯 역할
**데이터베이스 및 백엔드 API 제공**

### 📋 주요 기능

#### ✅ PostgreSQL 데이터베이스
- 방문 예약 정보 저장
- 사용자 정보 관리
- 공지사항, FAQ 등 콘텐츠 저장

#### ✅ 자동 API 생성
- 데이터베이스 테이블에 자동으로 REST API 생성
- 별도 백엔드 서버 개발 불필요
- SQL 쿼리를 HTTP 요청으로 변환

#### ✅ 실시간 기능
- 데이터 변경 시 실시간 업데이트
- WebSocket 연결 지원

#### ✅ 인증 (Authentication)
- 사용자 로그인/회원가입
- JWT 토큰 관리
- 소셜 로그인 지원

#### ✅ Row Level Security (RLS)
- 데이터 접근 권한 관리
- 보안 정책 설정
- 사용자별 데이터 접근 제어

#### ✅ 스토리지
- 이미지, 파일 저장
- 공개/비공개 파일 관리

### 💡 실제 사용 예시
```
1. 방문 예약 폼 작성
2. Vercel 배포된 웹사이트에서 제출
3. Supabase API로 데이터 전송
4. PostgreSQL 데이터베이스에 저장
5. 관리자가 Supabase 대시보드에서 확인
```

### 🔗 우리 프로젝트에서의 역할
- 방문 예약 데이터 저장
- 예약 번호로 조회
- 관리자 승인 기능
- 공지사항, FAQ 관리

---

## 🔄 세 서비스의 협업 흐름

### 1. 개발 단계
```
개발자 → 코드 작성 → GitHub에 푸시
```

### 2. 배포 단계
```
GitHub → Vercel이 자동으로 감지 → 빌드 → 배포
```

### 3. 사용자 접속
```
사용자 → 브라우저 → Vercel 배포된 웹사이트 접속
```

### 4. 데이터 처리
```
웹사이트 → Supabase API 호출 → 데이터베이스 저장/조회
```

### 5. 실시간 업데이트
```
Supabase 데이터 변경 → Vercel 웹사이트에 실시간 반영
```

## 📊 비교표

| 서비스 | 역할 | 비유 | 비용 |
|--------|------|------|------|
| **GitHub** | 코드 저장소 | 도서관 (책 보관) | 무료 (개인) |
| **Vercel** | 웹사이트 호스팅 | 전시관 (작품 전시) | 무료 (제한적) |
| **Supabase** | 데이터베이스 | 창고 (물건 보관) | 무료 (제한적) |

## 🎯 각 서비스가 없으면?

### GitHub 없으면?
- ❌ 코드 버전 관리 불가
- ❌ 협업 어려움
- ❌ 코드 백업 어려움
- ❌ Vercel 자동 배포 불가

### Vercel 없으면?
- ❌ 인터넷에 웹사이트 공개 불가
- ❌ 사용자가 접속할 수 없음
- ❌ 수동으로 서버 구축 필요

### Supabase 없으면?
- ❌ 데이터 저장 불가
- ❌ 방문 예약 정보 관리 불가
- ❌ 별도 백엔드 서버 개발 필요

---

## 💡 핵심 정리

1. **GitHub**: 코드를 저장하고 관리하는 곳
2. **Vercel**: 코드를 웹사이트로 만들어 인터넷에 공개하는 곳
3. **Supabase**: 웹사이트에서 사용하는 데이터를 저장하고 관리하는 곳

세 서비스가 함께 작동하여 완전한 웹 애플리케이션을 만듭니다! 🚀
